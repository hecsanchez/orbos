```
**Planning algorithm — implement in this exact order:**
```
1. Load student profile (age, grade_target, interests)
2. Load all mastery states for this student
3. Identify standards to work on:
   a. Standards with mastery_level < 0.8 (needs lesson or practice)
   b. Standards with no mastery record yet (never seen — needs lesson)
   c. Standards with mastery_level >= 0.8 (mastered — skip unless review)
4. For each candidate standard, decide type:
  - No attempts ever → type: lesson
  - Has attempts, mastery < 0.5 → type: lesson (reteach)
  - Has attempts, mastery 0.5–0.8 → type: practice
  - recommendation === 'practice' and no direct lesson attempt → type: lesson
5. Order standards:
  - Alternate subjects (never 2 Math in a row, never 2 Spanish in a row)
  - Prioritize lower mastery standards first within each subject
  - Mix grades if student is working across multiple grade levels
6. Insert breaks:
  - After every 20–25 minutes of lesson/practice, insert a break item (5 min)
7. Check for approved phenomena:
  - Query PhenomenonProposals where status = 'approved' and not yet completed
  - If found, append a phenomenon_evidence item at the end of the plan
8. Trim to fit available_minutes
9. Pre-fetch lesson scripts:
  - For each lesson/practice item, check if a script already exists in lesson_scripts
  - If yes, attach the script ID
  - If no, call LessonDesignerAgent or PracticeGeneratorAgent to generate one
  - This ensures the iPad can pre-fetch everything at session start
    Requirements:

No LLM call needed for the orchestration logic itself — pure TypeScript
The pre-fetch in step 9 may trigger lesson generation — this is intentional
If available_minutes is not provided, default to 120
Never plan more than 6 lesson/practice items per session — cap it
Always end with a break item if the last item is a lesson or practice
The plan must be deterministic for the same input — no randomness


Step 2 — Plan Caching
The iPad pre-fetches the daily plan at session start. Cache it so repeated calls don't re-generate scripts.
Create a daily_plans table migration:
sqlCREATE TABLE daily_plans (
id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
student_id uuid NOT NULL REFERENCES students(id),
date date NOT NULL,
plan_json jsonb NOT NULL,
generated_at timestamp DEFAULT now(),
UNIQUE(student_id, date)
);
Logic:

On GET /orchestrator/daily-plan/:studentId, check if a plan exists for today
If yes, return the cached plan
If no, generate a new one, cache it, return it
Add query param ?regenerate=true to force a fresh plan (useful for testing)
